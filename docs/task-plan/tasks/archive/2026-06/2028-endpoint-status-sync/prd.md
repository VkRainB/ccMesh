# 端点状态同步与模型获取优化

## Goal

消除端点管理页与仪表盘之间的状态变更延迟与不同步；让端点可用性显示遵循"实时请求结果优先于手动测试"；去除模型回落硬编码；通过输入辅助与鉴权聚合 + URL 候选策略大幅提高模型列表获取成功率。

## Requirements

1. 端点管理页启用/禁用端点后，仪表盘"启用端点"列表与计数及时更新（无需等待 staleTime 过期或手动刷新）。
2. 端点可用性徽章按优先级合并显示：实时请求得到的端点有效性 > 手动联通性测试结果。
3. 前端对健康事件的订阅与查询失效收口到共享 hook，消除两处逐字重复的订阅代码。
4. 模型拉取失败的回落不再硬编码单一 Claude 模型，按转换器格式给出对应默认模型，且默认模型表全局唯一（DRY）。
5. 端点表单 api_url 输入框下方实时展示按所选转换器类型拼出的完整请求 URL；检测到 `/v1` 结尾时给出警告提示（不自动修改用户输入）。
6. 表单"刷新模型"动作不再单纯依赖转换器类型决定唯一一种请求方式：先按所选类型的鉴权方式请求，失败后换另一种鉴权方式重试；仍失败才执行 URL 候选构造策略（剥离已知兼容子路径），任一步成功立即返回。

## Acceptance Criteria

- [ ] 在端点管理页 toggle 某端点，切到仪表盘，启用端点列表立即反映变化（事件驱动，非 60s 缓存过期）。
- [ ] 手动联通性测试完成后，仪表盘与端点管理页的健康/可用性显示同步刷新。
- [ ] 代理运行中，某端点实时请求成功（熔断器 healthy）时，即使其手动测试结果为 unavailable，可用性徽章也显示为正常；代理未运行或无实时数据时回退手动测试结果。
- [ ] ServiceCard 与 Endpoints 页不再各自内联订阅健康事件，统一经由共享 hook；事件触发时 `["endpoints"]`、`["health"]`、`["endpoint-health"]` 三个查询全部失效。
- [ ] 模型拉取回落默认模型按格式区分（Claude/OpenAI Chat/OpenAI Responses），项目中默认模型映射只存在一份。
- [ ] api_url 输入 `https://x.com/v1` 时表单出现警告提示；切换转换器类型时 URL 预览实时变化为对应后缀。
- [ ] 对形如 `https://api.deepseek.com/anthropic` 的端点，选 claude 类型且 x-api-key 鉴权拉模型失败时，自动用 Bearer 重试，再失败则按候选 URL（原始路径 → 剥离已知子路径）继续尝试，拿到模型列表即提前返回。
- [ ] `cargo test` 与前端 `tsc --noEmit`、`vitest run` 全部通过；URL 候选构造为纯函数并有单元测试。

## Definition of Done

- 上述验收标准全部满足。
- 改动不影响 `fetch_endpoint_models` 的 Tauri 命令签名（前端服务层零适配成本）。
- `get_models` 聚合路径行为保持现状（单次尝试），不被候选策略拖慢。
- progress.csv 子任务全部置完成，按模块 scoped 提交。

## User Stories

1. 作为用户，我希望在端点管理页开关端点后仪表盘立即更新，以便确认当前生效的端点集合。
2. 作为用户，我希望可用性显示以真实请求结果为准，以便不被过期的手动测试结论误导。
3. 作为开发者，我希望健康事件订阅收口到共享 hook，以便减少重复查询并降低维护成本。
4. 作为用户，我希望模型拉取失败时回落到与端点类型匹配的默认模型，以便 OpenAI 端点不会出现 Claude 模型名。
5. 作为用户，我希望输入 api_url 时看到完整请求 URL 预览与 /v1 结尾警告，以便第一次就填对地址。
6. 作为用户，我希望即使不清楚端点的真实协议类型，刷新模型也能通过鉴权聚合与 URL 候选自动试出正确组合，以便顺利拿到模型列表。

## Implementation Decisions

- **同步机制**：后端在端点更新（含启停）与手动联通性测试落库后 emit 新事件 `endpoints-changed`（空 payload，前端全量重拉）；前端新增共享 hook 统一订阅 `endpoints-changed` 与既有 `endpoint-health-changed`，事件触发时失效 `["endpoints"]`、`["health"]`、`["endpoint-health"]`。既有 mutation 的 onSuccess invalidate 保留作为即时兜底。
- **可用性优先级**：纯前端合并显示。可用性徽章优先消费实时健康查询中该端点的状态（healthy/recovering → 正常，unhealthy → 不可用），查不到该端点的实时数据或状态为 unknown 时回退 `testStatus`。不回写 DB，不动转发热路径。
- **默认模型表**：抽成按 `UpstreamFormat` 区分的单一映射函数，连通性测试与模型回落共用（现状两处各有一份，其中一处仅 Claude 硬编码）。
- **鉴权聚合**：所选类型对应的鉴权头先行（claude → x-api-key + anthropic-version；openai/codex → Bearer），空结果时换另一种鉴权同 URL 重试一次。
- **URL 候选策略**：仅在两种鉴权对原始 URL 均失败后启动。候选 = 原始 base + `/v1/models` → 剥离已知兼容子路径后 + `/v1/models`。明确**不做** `/v{N}` 结尾改 `/models` 这一档（由需求 5 的输入提示兜住）。每个候选 URL 同样按"先所选类型鉴权、再另一种"尝试，任一成功立即返回。已知兼容子路径清单（9 项）：`/api/claudecode`、`/api/anthropic`、`/apps/anthropic`、`/api/coding`、`/claudecode`、`/anthropic`、`/step_plan`、`/coding`、`/claude`。
- **策略作用范围**：聚合 + 候选策略仅在表单刷新命令路径启用；`get_models` 聚合命令维持现状单次尝试，避免遍历全端点 × 多候选 × 15s 超时的耗时放大。
- **/v1 结尾处理**：仅警告提示 + URL 预览，不自动剥离、不改用户输入（避免误伤真实路径含 /v1 的网关，且保证 JSON Tab 数据一致）。

## Testing Decisions

- URL 候选构造器实现为纯函数，覆盖单元测试：原始 URL、带尾斜杠、带已知子路径（含 DeepSeek 示例）、`/v1` 结尾、无可剥离子路径等用例。
- 默认模型映射函数单元测试覆盖三种格式。
- 前端跑 `tsc --noEmit` 与既有 vitest 套件；共享 hook 的订阅/失效行为以现有组件测试模式补测（可行时）。
- GUI 实际交互（toggle 后切页观察、表单提示显示）无法无头自动验证，显式声明并给人工核对清单。

## Out of Scope

- `/v{N}` 结尾改写为 `/models` 的候选档位（需求明确不做）。
- 实时请求结果回写 DB `test_status`。
- `get_models` 命令的前端接线（当前前端无人调用，保持现状）。
- api_url 保存时自动规整、后端入库规整。
- 熔断器阈值可配置化、后台健康轮询。
- 筛选 store 缺少 codex 选项的问题（调研顺带发现，另行处理）。

## Technical Notes

- `endpoint-health-changed` 事件仅在熔断器状态转换时发出且 payload 为空，前端依赖全量重拉；新事件沿用同一范式，保持一致性。
- 端点更新命令是通用更新入口（编辑表单同样走它），事件会在所有字段更新时触发——这正是期望行为（编辑后各页也应刷新）。
- 转换器为未知值（如 gemini）时后端按 Claude 处理，鉴权聚合天然覆盖该场景。
- 拉模型响应 Claude/OpenAI 上游同为 `data[].id` 结构，单一解析器即可。
